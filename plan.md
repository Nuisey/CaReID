# Plan: CarID Dashboard Web Application

This document outlines the architecture, features, and implementation steps for building the Flask-based web application to track neighborhood vehicles, integrating with your existing `CaReIDInference` pipeline.

## 1. System Architecture

Instead of reinventing the wheel, the Flask backend will act as a real-time monitor on top of your existing Re-ID pipeline.

*   **Backend (Python Flask)**: 
    *   Serves the UI (HTML, CSS, JS).
    *   Monitors `CaReIDInference\Data\CarLabels_Unprocessed.csv` for new vehicle detections.
    *   Serves vehicle images directly from `CaReIDInference\Data\Unconfirmed`.
    *   Maintains a persistent state (in `state.json`) for the customized house boxes and vehicle home/away statuses.
*   **Frontend (Vanilla JS/HTML/CSS)**: 
    *   **Homepage**: An interactive map using `houses.png` as the background. Allows creating and dragging "house" containers, managing car icons, assigning IDs, and visualizing real-time Home/Away states.
    *   **Timeline**: A real-time updating vertical feed showing recent vehicle detections, their images, identities, and whether they are arriving or leaving.

## 2. Core Features & Implementation Details

### A. Homepage (Interactive Map)
*   **Background**: `houses.png` will be copied to the Flask `static` folder to be served as the background of a designated map container.
*   **Customize Button**: Prompts the user (e.g., via a modal or native prompt) for a number of cars. It generates a draggable `<div>` containing that many car icons (using SVGs).
*   **Drag & Drop**: Implemented using HTML5 Drag and Drop or mouse-tracking events. The absolute/relative coordinates of each box relative to the background will be saved to `state.json` via a backend API.
*   **Car Assignment**: Clicking on a gray car icon will open a dropdown/modal populated with entries from `label_map.csv`. Selecting one binds that Re-ID to the icon.
*   **State Toggling**: Polling the backend every few seconds. If a car is "Home", the icon turns blue. If "Away", it turns gray.

### B. Timeline (Activity Feed)
*   **Feed Construction**: The backend will read `CarLabels_Unprocessed.csv` and parse the timestamp from the `filename` (e.g., `2025-11-08_12-41-24-474993__car_97858.jpg`).
*   **Display**: A scrollable vertical list. Each card will show:
    *   Image of the car (served by Flask from the `Unconfirmed` folder).
    *   Timestamp of detection.
    *   Vehicle Identity (Make, Model, Color, ID).
    *   Event Type: "Arriving" (Blue indicator) or "Leaving" (Gray indicator).

### C. Backend Storage (`state.json`)
A lightweight JSON file to persist configurations between reboots:

## 3. Approved Solutions & Refinements

Based on our discussion, here is how the technical challenges will be resolved:

1.  **Arriving vs. Leaving Logic**: 
    *   **Solution**: We will create a new, modified version of the YOLO script (`YOLO_Identification.py`). This script will track the vertical movement of the car's bounding box center over multiple frames.
    *   *Implementation Details*: If the `y` coordinate of the bounding box is increasing (moving downward), we classify the event as "Arriving". If the `y` coordinate is decreasing (moving upward), we classify it as "Leaving". We will encode this direction directly into the cropped image's filename (e.g., `2025-11-08_12-41-24__arriving__car_97858.jpg`), making it trivial for the ReID script and Flask backend to parse the exact state.
2.  **Image Parsing**:
    *   **Solution**: The backend will parse the exact timestamp directly from the filenames generated by the YOLO script. This guarantees exact chronological accuracy for the Timeline feed.
3.  **Project Location & Isolation**:
    *   **Solution**: The entire system will be completely self-contained within `C:\Users\nolan\OneDrive\Programming\CarID_Final`.
    *   *Implementation Details*: I will migrate all essential assets from your previous `CaReIDInference` project. This includes the model weights/config (`Brain/`), the gallery dataset, the label mapping, and core utility scripts (`model.py`, `load_model.py`, `dataset.py`). The main tracking and identification scripts will be re-written inside this new folder to directly power the Flask web app without relying on the old project directory.

## 4. Improvements

Per your request, I will implement the following two features:

1.  **Correcting a Label in the Timeline**:
    *   **Solution**: I will add an "Edit" button to the car label on each timeline card. Clicking it will open a search modal (just like the map's assign modal) pre-populated with all the vehicles from `label_map.csv`. 
    *   *Backend Logic*: When you submit a correction, the frontend will hit a new `/api/update_label` endpoint. The Flask backend will find the corresponding event in `CarLabels_Unprocessed.csv` by matching the exact filename, update the predicted ID and label in the CSV, and immediately refresh the timeline to show your correction.
2.  **Deduplicating Timeline Images**:
    *   **Solution**: Currently, YOLO takes a burst of pictures of a car while it's in the frame. I will update the backend timeline builder (`/api/timeline`) to group these bursts together. 
    *   *Backend Logic*: If the system sees multiple images of the exact same car ID going the exact same direction within a short time window (e.g., 60 seconds), it will filter them down to a single event. It will display the image that had the highest confidence score from that burst, keeping your timeline perfectly clean and concise!
